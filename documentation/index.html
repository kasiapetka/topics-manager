Project Specification:

System for handling projects topics. Application is a REST API created with Spring Boot Framework, React Library and with PostgreSQL database. It helps in organizing project groups and sections and managing them. Users appear in three different roles, there is one administrator who can add various teachers. Students can create account with special codes for authentication. Specific requirements and features are shown on multiple diagrams and charts below. Please feel encouraged to get acquainted with the application and its documentation as it may serve You as an instruction for using it.

Workflow

We are trying to use Agile Developement. We are making a lot of meetings where we discuss what is the best for our project. We are trying to follow our Gantt Chart and sometimes we have to make decisions to leave some ideas behind in order to get most important functionality working.

Unfortanently this software isn't made in TDD technique, however we're trying to follow desing patterns and create clean code which is sometimes not that simple. After implementing bigger part of functionality we are reading our code and trying to make it cleaner. We are removing code duplication, splitting some functions into shorter ones or renaming functions, variables, classes etc to follow naming conventions.

Stage One

In stage one we were discussing how this project will work. We have specified project requirements and then we have splitted work between all team members. With these informations we were able to create gantt's diagram. After that we also created use case diagram for our project

In second week of phase one we have set up project's structure and github repository. After that we have started thinking how or model should look like. We were consulting our ideas with people with greter expierience and after getting satisfying result we implemented it into our project with usage of JPA. With that done we created first class diagram

The last part of stage one was creating simple logic for registration and login. We decided that we want to use JWT for that purpose. Unfortanently JWT was completely new and it took us a lot more time that we thought so. So in this stage we have set up login and registration form and some requests for database but we didn't make that in time as our gantt's chart suggests.

Stage Two

We have began stage two from setting up bigger meeting and discussing what we should do next. Due to our long learning process finishing registration and login for every user took us some time.

In week two of second stage we have began implementing full account modification for every type of user. We have started writing first Services, more controllers and our project started to look like MVC pattern. We have skipped implementing code authentication for student's registration because we decided that this is not crucial for this app to work and we have continued to implement account modification.

At this moment we have started to notice that we are not following gantt's chart because our assumptions weren't correct. Nevertheless in the end of stage two we have spent a lot of time refactoring our code and we created second class diagram.

Stage Three

In stage three we were focused on admin's and teacher's options. Most of our time we were working on editing teacher's accounts by admin, adding them or removing them. When we had that done we have started to add more admin functions like creating subjects editing them or removing.

We realised that we are implementing things too slow. We were at the end of stage three but we haven't even finished stage two. The reason behind that was that we were more focused on creating clean code and trying to implement good solutions by sometimes talking for hours about one not that important thing instead of making app work.

Stage Four

In stage four in first week we have implemented codes authentication when student tries to register, full modification of subject and adding teachers to it by admin. Also we have finished implementing topics and its functionality. We have created management system for sections by admin and teacher and we linked these three components (subject, topic, section) togheter.

In second week of phase four we refactored our gantt's chart because we were falling apart. We managed to finish basic implementation of issuing grades. We have made a lot of refactoring on front-side for better code transparency. We have also speed up with work due to our delay.

Spring MVC in Topics Manager

This applications is built around Spring MVC Framework, which means that we are explicitly separating Model, View and Controller part of application.

ServiceLayer

Since this application uses Spring MVC framework and its capability of using dependency injection and inversion of control ServiceLayer has been created.

Service Layer specifies interfaces with functions which are neccesary for application to work. Then they these functions are implemented and injected into placere where they are needed. Thanks to Spring's Dependency Incjection it's very easy to implement different functionality and incject it.

Authentication and Authorization

Authentication and authorization is achieved with using two main components: Spring Security and JWT. In Topics Manager every User that is registered into system has login which is email, and password. Aditionally every user has its role: Admin, Teacher or Student.

Spring Security

Spring Security is configured to listen for requests on /api/login endpoint. When it gets correct request it is performing two queries for database. First query checks if given credentials were correct and second is checking which role this user has.

Authorization using Json Web Tokens:

Authorization in Topics Manager Application is implemented using Json Web Tokens. When the user wants to log in and passes correct credentials server responds with a token that holds user email and the time of token expiration. Then whenever the user wants to retrive information from the server client app passes the token he recieved in a HTTP request header. This way of authentication brings the application security on the highest level.

Users

Every user is stored in the same table called "users". Users consist of email, password, id and role which is assigned when user is registered. Every user is assigned to Student or Teacher object except admin who doesn't have name, surname etc.

This structure of tables in database makes it easy for searching and making operations only on accounts or only on teacher or stundent objects. Also this operation was neccesary because there is possibility that Student exists in database without account.

Accounts

In topic Manager there are three types of accounts: admin account, teacher account and student account. Every type of account has different role and different capabilities of managing the system

Admin Account

An Admin Account is only created together with the database. There is no option of adding another Admin when the application is running.

Admins functionalities consist of
  • Adding, deleting and modifying a Teacher account
  • Adding, deleting and modifying a Student account
  • Modifying his own account
  • Adding a new Subject
  • Modifying list of Teachers that teach a particular Subject

    • Teacher Account

    A Teacher is responsible for issuing presence and grades to Students. He can create his own Topics, other Teachers can join and his own Sections other Teachers can not join.

    Teachers functionalities consist of
  • Modifying his own account
  • Joining a Topic

    • Admin and Teacher Accounts common functionalities
  • Adding a new Topic to an already existing Subject
  • Adding and deleting a Section
  • Adding and editing Students in his Sections
  • Opening, closing or terminating a Section

    • Student Account

    Student can use all functionalities connected with sections. He participates in different sections in various topics, this way he can be up to date with everything important happening inside a section he is a member of. He can join only one section in a given topic. If he wants to reassign he will be informed of leaving the section he is currently in.

    Student functionalities consist of
  • Joining/leaving sections
  • Listing sections on all semesters
  • Viewing section information, grade and presence
  • Adding attachments

    • Sections

    Sections are instances in a given subject, topic and with a unique name guided by one teacher. Teachers add and manage members of sections, they can issue presence and grades. Sections have different states in which students can or cant join them. Students can add attachments to sections and view information.

    Subject

    The most general division of issues that the section can implement. Subjects consist of topics. They can be added only by admin because we usually have only about a dozen of them. Admin can add them as well as list and filter them. Also teachers are assigned to subjects, and they can create sections only in their subjects. The admin can add, edit and delete teachers assigned to subjects.

    Topic

    Topics create a more precise division of issues in a given subject. They are assigned to subjects. Admin can add topics in all subjects, teachers can add topics only in subjects they are a part of. Admin and teachers can also list and filter topics.

    Messaging System

    Messaging system is used for sending messages between users. After logging in user will have notification about new messages. Message is considered as new when it hasn't been read. Message can be read by clicking on it. Messages also can be removed, by removing message from inbox or from sent panel it will be removed only for the perso who requested it. Also there is automatic message sending when teacher tries to add student to section which requires leaving the one he's already in.

    Summary

    In conclusion the project was completed within a given amount of time, it realizes all functionalities and assumptions that we wanted it to have. Thing we would like to improve in the future is messaging system, because it could be a socket listening to new messages and we could notify the user at the time he gets a new message about it. The rest is done pretty well, the application is intuitive and really nice to use. We had an opportunity to learn many new technologies and improve our coding skills. Summarizing we are pleased with the effect and we hope we could expand this version easily in the future.
